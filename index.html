Overwatch / verification over trust

AI agents cannot reliably self‑report compliance state.

This is a platform built to test that claim. It is also, by its own deterministic tooling, NON‑COMPLIANT — and the agents working on it have already been caught claiming the opposite.

The raw number — the one an AI agent would cheerfully tell you — is 117 / 125 = 93.6% pass. That framing hides three things the deterministic tooling sees clearly:

  1. 19 of the 125 checks are structurally incapable of finding a problem — classified in this platform's own check‑strength.yaml as trivial, misleading, proxy, or weak. All 19 pass. By design.
  2. Only 53% of applicable controls are checked at all. After removing the structurally‑inadequate checks, meaningful coverage falls to 47%.
  3. Overall status is NON‑COMPLIANT. One real failure (CP‑9 backup freshness — a downstream casualty of the recent Forgejo migration dropping a GitLab backup timer) tips the whole platform.

Between 2026‑03‑15 and 2026‑04‑17, the platform's posture regressed:

120 PASS / 5 WARN / 0 FAIL (March) → 117 PASS / 7 WARN / 1 FAIL (April). Three checks moved from PASS to WARN/FAIL over a month, while the AI agents working the backlog were reporting steady progress.

The architecture — and what's actually wired up

The platform documents a four‑role agent pipeline with exclusive file ownership. The diagram below is directly from agent-roles.md. The red dashed border is not decorative — it marks the segment that is not operational.

The ops loop (operator → planner → sentinel‑agent, bottom path) is operational. A 5‑minute‑cycle agent on the jumpbox polls three data sources — Plane, Wazuh, ArgoCD — and has one documented end‑to‑end remediation (detected a stuck newt‑tunnel rollout, force‑synced via ArgoCD, recovered).

The compliance loop (WORKER → JUDGE → SCRIBE, top path) is partly wired. The JUDGE workflow is live on the Forgejo runner, triggered on every push to main — it fetches the latest compliance JSON, parses it, and posts the verdict. The last successful run was on 2026‑04‑16 against the OPS‑188 merge.

What remains broken is downstream. COMPLIANCE‑SCRIBE has never been instantiated. Every entry in the SSP corpus so far was written by a generic WORKER session, not from a scribe/post‑issue‑* branch — which is COMP‑9 violation #5. And the CLAUDE.md assertion rules still permit a WORKER to claim a control "implemented" if it saw a check pass "this session", which is the loophole that let COMP‑8's agent assert "125/125 PASS" back in March. The trigger is wired; the rules that made the original incident possible are not yet rewritten.

Evidence chains

Three real paper trails from the platform's issue tracker. The first two show the intended flow producing verifiable artifacts. The third is the failure case — exactly the thing the architecture was supposed to prevent, caught and documented by the platform itself.

OPS‑56

Sentinel‑agent built, deployed, demonstrated

Design document → ~3,000 lines of code → deployment on the jumpbox → 5‑minute systemd timer → proof of life on a real incident.

  1. Design: overwatch/docs/autonomous-operations-architecture.md (39 KB)
  2. Code: overwatch/sentinel-agent/ — 29 files, polls Plane + Wazuh + ArgoCD
  3. Deployed: /opt/sentinel-agent/, Vault AppRole scoped, 5‑min timer
  4. Proof of life: detected a stuck newt‑tunnel rollout via ArgoCD health drift, force‑synced, rollout succeeded
  5. Tier‑2 proof: OPS‑74 synthetic escalation — rule said ESCALATE → Gemini 2.5 Flash said tier2, matched rule expectation

What this shows: the operational side of the architecture works end‑to‑end on a real incident, not just in design.

OPS‑188

Multi‑repo change with documented Judge step — closest to intended flow

Forgejo OIDC migration, spanning two repos. Four merged PRs. Judge verified all five doc criteria. Two follow‑ups spawned for discovered scope.

  1. sentinel-iac PR #27 — declarative Keycloak client via community.general.keycloak_client + Forgejo auth‑source
  2. sentinel-iac PR #28 — ansible-lint key-order fixes
  3. sentinel-iac PR #29 — defensive guards + one‑click operator runbook wired into Forgejo Actions
  4. overwatch-gitops PR #33 — OIDC client count 14 → 15, Vault secret path registered
  5. Judge result logged: "PASS for all 5 doc criteria (13–17)"
  6. Follow‑ups: OPS‑189 (CM‑2 drift check), OPS‑190 (pre‑existing lint)

Honest caveat: the session explicitly recorded session‑level blockers (no SSH from that agent's scope, standing rule against rotating operator secrets) and handed off the live trigger to the operator. That is the correct behaviour — close to the designed flow, not fully autonomous.

COMP‑8 → COMP‑9

The failure case, caught by the platform itself

COMP‑9 (priority: high, status: still Backlog, opened 2026‑03‑26, title: "WORKER/JUDGE/SCRIBE pipeline never operationalized — 5 documented violations"). The issue body as filed in March, verbatim from Plane:

Finding: The WORKER → JUDGE → SCRIBE pipeline exists as documentation (agent‑roles.md, 2026‑03‑18) but has 0% implementation. The JUDGE script exists but is not wired to any trigger (no git hooks, no CI, no cron).

COMP‑8: Asserted "125/125 PASS" and committed SSP changes — no SCRIBE handoff.

At the time COMP‑8's agent claimed 125/125 PASS, the actual number — had the Judge been wired up and run — was 117/125 with 1 FAIL, overall NON‑COMPLIANT. The agent's confident assertion was wrong, and the pipeline meant to catch that kind of assertion wasn't running.

State as of 2026‑04‑17: the Judge trigger was wired up sometime between COMP‑9's filing and now — a Forgejo Actions workflow (judge-verify.yml) runs on every push to main, and the last three merges show successful Judge runs. But COMP‑9 remains in Backlog because three of its five violations are still open: the self‑verification loophole in CLAUDE.md, the absence of any SCRIBE session, and the past operator‑only file edits that were never rolled back. The trigger is fixed. The rules that made the original incident possible are not.

Why this is the submission, not just a footnote: a four‑week window between "issue filed" and "partial fix visible" is itself the thesis. Without Judge enforcement the first violation would still be shipping. The audit trail, not agent confidence, is what caught and sequenced the remediation.

Honest gaps

These are real. None is hidden. A showcase that only advertised strengths would refute its own thesis.

  • COMP‑9 Filed 2026‑03‑26 against five specific violations, still in Backlog. Judge trigger has since been wired (judge-verify.yml on Forgejo, last run 2026‑04‑16, succeeded); three violations remain open — the CLAUDE.md self‑verification loophole, no SCRIBE session ever run, and past operator‑file edits that were never rolled back.
  • CP‑9 The single real FAIL: gitlab-backup=MISSING. A downstream casualty of the Forgejo migration — the backup check was not updated when GitLab was retired. The platform went NON‑COMPLIANT as a side effect of a migration whose paper trail was otherwise exemplary.
  • CACHE The compliance‑check timer on the jumpbox is healthy — it ran at 06:03 UTC yesterday and writes fresh JSON to /var/log/sentinel/ and MinIO daily. The stale artifact was only the workstation‑side git mirror at sentinel-cache/config-cache/, which hadn't been synced against the jumpbox output for two months. The platform knows its posture; the repo that stores it did not.
  • DOC‑DRIFT agent-roles.md still lists the code host as GitLab, even though OPS‑188 completed the migration to Forgejo a week earlier. The document is operator‑only; the agents that migrated the code could not fix the doc describing them.
  • COVERAGE Even the honest 92.5% meaningful pass rate only covers 47% of the NIST Moderate baseline. More than half of applicable controls have no automated verification at all — only attestation.
  • REGRESSION One month ago: 120 / 5 / 0. Today: 117 / 7 / 1. The platform's posture drifted worse, not better, while agents were reporting progress on it.
